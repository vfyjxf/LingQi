# AGENTS.md

本文件是 LingQi 项目的项目级协作规范。后续所有人类开发者和 AI Agent 在本仓库内工作时，都应优先遵守这里的约定。

## 1. 沟通与提交语言

- 所有面向项目的说明、提交信息、PR 标题和 PR 描述默认使用中文。
- 技术名词可以保留英文，例如 Next.js、Pull Request、Review、API、schema。
- 回答用户时保持直接、具体，少写空泛描述。

## 2. 项目目标

LingQi 是一个本地运行的 AI Pull Request Review 助手。

当前 MVP 的目标是：

- 用户输入 GitHub Pull Request URL。
- 系统自动获取 PR 元信息、文件变更、提交记录和 diff。
- 系统生成结构化 Review 报告。
- 报告包含 PR 变更总结、审查重点、风险识别、Review 建议和上下文说明。

不要把当前 MVP 扩大成完整 GitHub App、PR 机器人、企业治理平台或通用聊天助手。这些可以写在未来计划中，但不作为当前核心实现范围。

## 3. 项目有效性要求

本项目需要保持清晰、可追溯、可运行的开发过程：

- 项目实现必须始终围绕核心方向：AI PR Review 助手。
- 代码应自主完成，不抄袭。
- 开发过程要持续交付，避免临近交付时一次性导入大量代码。
- commit 时间应真实反映开发过程，不伪造、不集中补录。
- 每个 PR 的标题和描述必须清晰，不能空白，也不能与实际代码变更严重不符。
- 引用第三方库或框架时，必须在 README 中列明依赖，并说明本项目的原创功能部分。
- 如果复用了自己过去的代码片段，必须在对应 PR 描述中注明来源和复用范围。

## 4. PR 规范

开发新功能时基于 PR 进行，不直接把大功能一次性堆到主分支。

每个 PR 尽量只做一件事：

- 一个基础设施调整。
- 一个数据解析能力。
- 一个 API 能力。
- 一个 UI 组件或页面区域。
- 一个测试补充。
- 一个文档更新。

PR 描述必须包含：

- 标题：一句话说明本 PR 新增或修改了什么。
- 功能描述：说明功能作用和使用方式。
- 实现思路：说明技术选型或核心逻辑。
- 测试方式：说明如何验证功能正常运行。
- 复用说明：如果复用了历史代码，说明来源和范围；没有复用则写明没有复用历史代码。

PR 合并后，主分支必须保持可运行。其他开发者在任意时间查看主分支，都应该能按 README 启动项目。

## 5. 提交规范

提交信息使用中文，建议格式：

```text
feat: 实现 PR URL 解析
fix: 修复 GitHub API 错误提示
docs: 补充本地运行说明
test: 增加报告 schema 校验测试
chore: 初始化应用框架
```

提交应保持小而清晰。不要把无关修改混进同一个提交。

## 6. 第三方依赖与原创说明

新增第三方依赖时必须同步检查 README：

- README 中是否列出了该依赖。
- 是否说明该依赖在项目中负责什么。
- 是否说明项目原创功能和第三方能力的边界。

不要把第三方库、示例代码或模板代码描述成本项目原创功能。

当前项目原创重点包括：

- GitHub PR URL 解析与校验。
- GitHub PR 数据获取与归一化。
- PR 分析上下文构建。
- 风险提示与 Review 建议结构设计。
- AI Review 编排逻辑。
- 结构化报告 UI。
- 本地 demo/mock 分析模式。

## 7. 本地文件与提交边界

- 不要提交 `docs/` 目录。该目录用于本地调研、设计和计划记录。
- 不要提交 `.env`、`.env.local` 或任何密钥。
- 不要提交 `.next/`、`node_modules/`、`tsconfig.tsbuildinfo` 等生成物。
- 暂存文件时优先使用精确路径，避免误用 `git add .`。

## 8. 技术栈边界

当前技术栈：

- Next.js
- React
- TypeScript
- Tailwind CSS
- Zod
- Vitest
- GitHub REST API

除非有明确收益，不要引入大型状态管理库、复杂 UI 组件库、数据库、登录系统、GitHub App、Webhook 或云部署流程。

如果确实需要新增技术，先说明：

- 它解决什么问题。
- 为什么现有技术不够。
- 对两天内完成 MVP 是否有风险。
- README 需要如何更新。

## 9. 代码质量要求

- 优先写小而清晰的模块。
- 纯逻辑放在 `lib/`，UI 放在 `components/` 或 `app/`。
- API Route 只做请求校验和编排，不堆积复杂业务逻辑。
- AI 输出必须经过结构化 schema 校验。
- 错误信息要能帮助用户理解问题，例如 PR URL 无效、GitHub API 限流、模型密钥缺失。
- 不要用 AI 生成大量低价值 Review 评论；产品设计应强调低噪音、高置信、有证据。

## 10. 验证要求

提交前根据改动范围运行必要检查：

- 修改 TypeScript 代码后运行 `npm run typecheck`。
- 修改核心逻辑后运行 `npm test`。
- 修改页面或路由后尽量运行 `npm run build`。

如果某个检查无法运行，必须在最终说明或 PR 描述中写清楚原因。

## 11. README 维护要求

README 必须持续反映项目真实状态：

- 当前项目能做什么。
- 如何安装依赖。
- 如何本地启动。
- 使用了哪些第三方依赖。
- 哪些部分是本项目原创实现。
- 是否存在历史代码复用。

当功能、依赖、运行方式发生变化时，同步更新 README。
